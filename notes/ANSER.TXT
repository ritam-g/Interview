1.Aggregation Pipeline in MongoDB is used to process data step by step. It works like a pipeline where data goes through different stages, and each stage does a specific task.

We use it for:

* filtering data
* grouping data
* sorting data
* calculating values
* joining collections

For example, in a shopping app, we can use aggregation to find total sales or top-selling products.

Some common stages are:

1. $match → filters data
2. $group → groups data and performs calculations
3. $sort → sorts data
4. $project → selects specific fields
5. $limit → limits the results
6. $lookup → joins another collection

In simple words, aggregation pipeline helps us analyze and transform data easily inside MongoDB.



2. Indexing in MongoDB is used to make data searching faster. Without an index, MongoDB checks every document in the collection to find data, which takes more time.

An index works like a book index. Instead of reading the whole book, we directly find the page number using the index.

For example, if we search users by email many times, we can create an index on the email field. Then MongoDB can find the user much faster.

Compound index means an index created on multiple fields together.

Example:

```js
{ name: 1, age: 1 }
```

This index helps when searching or sorting using both name and age fields together.

In simple words:

* Single index → works on one field
* Compound index → works on multiple fields

3.What’s the difference between find(), findOne(), and aggregate()?

find() is used to get multiple documents from a collection.

Example:

```js id="wqdvye"
User.find({})
```

It returns all matching documents in an array.

---

findOne() is used to get only one document.

Example:

```js id="v4m1uv"
User.findOne({ email: "test@gmail.com" })
```

It returns the first matching document.

---

aggregate() is used for data processing and analysis. It works with stages like filtering, grouping, sorting, and calculations.

Example:

```js id="7w6q4r"
User.aggregate([
  { $match: { age: { $gt: 18 } } },
  { $group: { _id: "$city", totalUsers: { $sum: 1 } } }
])
```

It is mainly used for complex queries and reports.

In simple words:

* find() → get many documents
* findOne() → get one document
* aggregate() → process and analyze data

4.How do you optimize MongoDB queries for performance?

To optimize MongoDB queries for better performance, we mainly try to reduce the amount of data MongoDB needs to scan.

Some common ways are:

1. Use Indexing
   Indexes make searching faster. Frequently searched fields like email, username, or id should have indexes.

2. Use Proper Queries
   Only fetch required data instead of getting all fields.

Example:

```js id="2qk8kh"
User.find({}, { name: 1, email: 1 })
```

3. Use limit()
   Limit the number of documents returned.

4. Avoid Unnecessary Data
   Do not store very large unnecessary data in one document.

5. Use Aggregation Properly
   Use $match early in the aggregation pipeline to filter data first.

6. Use Pagination
   Instead of loading all data at once, load data page by page.

7. Monitor Query Performance
   MongoDB provides explain() to check how queries are working.

In simple words, MongoDB performance is improved by using indexes, writing efficient queries, and reducing unnecessary data processing.

5. What are the different types of relationships in MongoDB?


## 1. One-to-One (1:1) Relationship

One document is linked with only one other document.

Example:

* User ↔ Profile

We can store it in two ways:

* In same document (embedding)
* Or separate collections with reference

Example:

```js id="1kq9ab"
User {
  name: "Ritam",
  profile: {
    age: 21,
    city: "Kolkata"
  }
}
```

---

## 2. One-to-Many (1:N) Relationship

One document is linked with many documents.

Example:

* One user → many posts

We usually use:

* embedding (small data)
* or referencing (large data)

Example:

```js id="7kq2ld"
User {
  name: "Ritam",
  posts: [postId1, postId2, postId3]
}
```

---

## 3. Many-to-Many (N:N) Relationship

Many documents are related to many other documents.

Example:

* Students ↔ Courses

We use referencing with arrays on both sides or a separate mapping collection.

Example:

```js id="9xq8dd"
Student {
  name: "A",
  courses: [courseId1, courseId2]
}

Course {
  title: "Math",
  students: [studentId1, studentId2]
}
```

---

## Simple Interview Summary

Schema design in MongoDB means deciding how to structure data. We can model relationships in three ways:

* 1:1 → single embedded document or reference
* 1:N → array or references
* N:N → references or mapping collection

In simple words, schema design depends on how data is related and how we want to access it efficiently.


6.Explain data validation and how to enforce it in MongoDB.


Data validation in MongoDB is used to make sure only correct and valid data is stored in the database. It helps prevent wrong data types, missing fields, or invalid values.

In Node.js applications, we commonly enforce validation using Mongoose schemas. Enforcing validation means applying rules to the data before saving it into MongoDB. If the data does not follow the rules, Mongoose throws a validation error and does not save the data.

For example:

```js id="u4t7bn"
const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: true
  },
  email: {
    type: String,
    required: true
  },
  age: {
    type: Number,
    min: 18
  }
})
```

In this schema:

* name is required
* email is required
* age must be at least 18

If a user tries to save invalid data, Mongoose blocks it and returns an error.

Data validation is important because it keeps the database clean, secure, and consistent.


7.What are transactions in MongoDB and when would you use them?

Transactions in MongoDB are used when we want multiple database operations to happen safely together.

It works on the rule:

> Either all operations succeed or none of them succeed.

This is called the “all or nothing” concept.

---

For example, think about a banking app.

Suppose:

* ₹1000 is transferred from User A to User B

Two operations happen:

1. money is removed from A’s account
2. money is added to B’s account

Now imagine:

* money is removed from A
* but server crashes before adding money to B

This creates incorrect data.

To solve this problem, we use transactions.

If any operation fails:

* MongoDB cancels all changes
* database stays correct and consistent

---

Example flow:

* Start transaction
* Perform multiple operations
* If everything works → commit transaction
* If something fails → rollback transaction

---

Simple example:

```js id="q2m8vc"
const session = await mongoose.startSession();

session.startTransaction();

try {
  // operation 1
  // operation 2

  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
}
```

---

We use transactions when:

* payment systems
* banking apps
* order management
* wallet systems
* booking systems

where data accuracy is very important.

In simple words:

> Transactions help MongoDB perform multiple operations safely together without losing data consistency.

## **8. What is a Capped Collection?**

A capped collection in MongoDB is a special type of collection that has a fixed size.

When the collection becomes full:

* MongoDB automatically removes the oldest documents
* and inserts new documents in their place

So, it works like a **circular queue (FIFO system)**.

---

### **Example (Easy Understanding)**

Think of a CCTV recording system:
Old recordings are automatically deleted when storage is full.
Similarly, capped collections automatically remove old data.

---

### **Main Features**

* Fixed size collection
* Fast insertion
* Maintains insertion order
* Automatically deletes oldest data

---

### **Use Cases**

* Logs
* Chat messages
* Notifications
* Real-time monitoring data

---

### **How to Create in MongoDB (Proper Way)**

```js id="capmongo1"
db.createCollection("logs", {
  capped: true,
  size: 100000
})
```

---

## **Can we define capped collection in Mongoose schema?**

✔ Yes, but only in some cases
✔ It can be defined in schema options
✔ But the recommended and reliable way is collection-level creation

---

### **Schema-level (Sometimes works)**

```js id="capschema1"
const logSchema = new mongoose.Schema(
  {
    message: String
  },
  {
    capped: { size: 100000 }
  }
);
```

---

### **Recommended (Production Way)**

```js id="capbest1"
await mongoose.connection.createCollection("logs", {
  capped: true,
  size: 100000
});
```

---

### **Important Interview Point**

* Schema = defines structure of data
* Capped collection = defines behavior of collection (storage rule)

So capped collections are mainly a **MongoDB collection-level feature**, not schema-level.

---

## **Final Simple Interview Answer**

A capped collection is a fixed-size MongoDB collection where old data is automatically removed when new data is inserted. It is used for logs and real-time data.

We can create it using `capped: true` at collection creation time, and in Mongoose it is recommended to define it at the collection level rather than only inside schema.

9.How does MongoDB handle concurrency?

## **9. How does MongoDB handle concurrency?**

MongoDB handles concurrency by allowing multiple users to read and write data at the same time without blocking the entire database.

It uses **document-level locking**, which means when one document is being updated, only that specific document is locked, not the whole collection or database. This allows other documents to be accessed and modified at the same time.

MongoDB also uses a technique called **MVCC (Multi-Version Concurrency Control)**. This allows readers to access a stable version of data while writers are updating it, so reads and writes can happen together without blocking each other.

In addition, MongoDB uses **transactions** when multiple operations need to be executed together safely. This ensures data consistency in complex operations.

---

### **In simple words:**

MongoDB handles concurrency by locking only the required document and allowing multiple read and write operations to happen at the same time using MVCC, which improves performance and scalability.


10.Explain change streams in MongoDB.

## **Explain Change Streams in MongoDB**

Change Streams in MongoDB are used to **listen to real-time changes** happening in a collection, database, or even an entire MongoDB cluster.

It allows an application to get notified whenever data is:

* inserted
* updated
* deleted

without repeatedly checking the database.

---

## **How it works**

MongoDB continuously tracks changes in the background.
Whenever any change happens, it sends an event to your application in real time.

This is done using the `.watch()` method.

---

## **Example (Mongoose)**

```js id="cs1"
const changeStream = User.watch();

changeStream.on("change", (change) => {
  console.log("Change detected:", change);
});
```

---

## **Where it is used**

Change Streams are used in real-time applications like:

* Chat applications (new messages)
* Notification systems
* Live dashboards
* Activity tracking systems

---

## **Important Note**

Even if we use **Mongoose models**, Change Streams work because:

* Mongoose models are connected to MongoDB collections
* `.watch()` listens directly to the underlying MongoDB collection

---

## **Simple Working Flow**

```text id="cs2"
Application (Mongoose Model)
        ↓
MongoDB Collection
        ↓
Change Stream (.watch)
        ↓
Real-time event sent to app
```

---

## **In simple words:**

Change Streams in MongoDB allow the application to **listen and react to real-time database changes** without polling or repeatedly querying the database.

---

## **Final Interview Answer**

Change Streams in MongoDB are used to listen to real-time changes in a collection or database. Using `.watch()`, the application can detect when data is inserted, updated, or deleted and respond immediately. It is mainly used in real-time applications like chat apps, notifications, and live dashboards.
